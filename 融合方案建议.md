# loop · OpenSpec · Superpowers 三项目融合方案

> 三个项目分属不同层次，合在一起恰好形成 AI 驱动开发的完整链路。

---

## 一、现状定位

```
┌────────────────────────────────────────────────────────────────┐
│                         OpenSpec                                │
│              AI-native 规约层 — 定义"做什么"                      │
│  openspec/specs/ → openspec/changes/ → delta specs              │
│  结构化 Given/When/Then 规格，版本化变更管理                       │
│  CLI: openspec init/update/validate/archive                     │
├────────────────────────────────────────────────────────────────┤
│                         loop                                    │
│              自动化执行管线 — 把"做什么"变成"怎么做"                │
│  PRD → SPEC → Issues → Tasks → 自动执行 + 重试 + 提交            │
│  Git 跟踪、结构化报告、AC 验证                                    │
│  CLI: prd_pipeline.sh / task_executor.sh / progress.sh          │
├────────────────────────────────────────────────────────────────┤
│                       Superpowers                               │
│              AI 开发方法论 — 定义"怎么工作"                        │
│  14 个 Skill: brainstorm → plan → TDD → review → finish         │
│  质量门禁、流程管控、跨 Agent 协作                                 │
│  加载方式: hooks/session-start 注入 context                     │
└────────────────────────────────────────────────────────────────┘
```

三者不重叠，定位互补：

| 维度 | OpenSpec | loop | Superpowers |
|------|----------|------|-------------|
| 回答的问题 | 做什么 | 怎么做 | 怎么工作 |
| 输入 | 开发者的意图 | 规格文档 | 任意任务 |
| 核心产出 | 结构化 specs | 可执行 tasks + 代码 | 流程合规 |
| 执行方式 | CLI + 斜杠命令 | shell 自动化管线 | Agent context 注入 |
| 状态管理 | 文件系统 specs | 文件系统 + git | Agent 会话 |

## 三、融合方向（由浅入深）

### 方向 A：loop 注册为 Superpowers Skill（推荐起步）

**目标：** 让 loop 的能力可从 Claude Code 内直接召唤。

**做法：** 在 `superpowers/skills/` 下新增 skill 文件：

```
superpowers/skills/
├── prd-pipeline/
│   └── SKILL.md          ← "从 PRD 到代码的全自动管线"
├── task-executor/
│   └── SKILL.md          ← "执行/重试 Task 文件"
└── review-report/
    └── SKILL.md          ← "查看执行报告"
```

**`prd-pipeline/SKILL.md`** 示例内容：

```markdown
---
name: prd-pipeline
description: 从 PRD 到 Task 拆解到代码执行的完整管线
---

当用户说"跑一下 PRD 管线"或"/prd-pipeline"时，执行以下步骤：

1. 确认 PRD.md 存在：
   ls prd-pipeline/PRD.md

2. 执行全流程（项目模式）：
   cd prd-pipeline && ./prd_pipeline.sh PRD.md --target-dir $(pwd) --granularity fine --direct

3. 如果用户指定了参数（如 `--skip-to execute`），追加到命令后。

4. 完成后展示进度：
   cd prd-pipeline && ./progress.sh

5. 如果有 summary.md，展示给用户：
   cat prd-pipeline/projects/*/04-execution/summary.md 2>/dev/null || echo "无报告"
```

**工作量：** 半天。只需创建 3 个 SKILL.md 文件。
**收益：** loop 的能力从"手动运行脚本"变成"在 Claude Code 里斜杠命令召唤"。

---

### 方向 B：loop 消费 OpenSpec Specs

**目标：** 用 OpenSpec 的结构化 specs 替代 loop 的自由格式 PRD，跳过 PRD→SPEC 步骤。

**当前 loop 的链路：**

```
PRD.md（自由格式）
  → prd_to_spec.sh（调用 Claude 分析生成 SPEC.md）
    → spec_to_issues.sh（再调用 Claude 拆解 Issues）
```

**融合后的链路：**

```
openspec/specs/（结构化规格，已含技术设计）
  → issue_to_tasks.sh --spec-dir openspec/（直接拆解 Issues，跳过前两步）
    → task_executor.sh（执行不变）
```

**具体改动：**

1. `issue_to_tasks.sh` 新增 `--spec-dir` 参数
2. 读取 `openspec/specs/` 目录下的 MD 文件，合并为 SPEC 上下文
3. 读取 `openspec/changes/<active>/tasks.md` 如果有，直接作为 task 定义
4. 跳过 `prd_to_spec.sh` 和 `spec_to_issues.sh`，节省两次 Claude 调用

**当 OpenSpec 没有 tasks.md 时（更常见）：**

```bash
# loop 读取 specs/ 作为 SPEC 上下文
./issue_to_tasks.sh openspec/specs/ ISSUES.md --spec-dir mode

# 然后正常执行
./task_executor.sh
```

**当 OpenSpec 有 tasks.md 时（最轻量）：**

```bash
# loop 跳过拆解，直接把 tasks.md 转为 task 文件
./issue_to_tasks.sh openspec/specs/ openspec/changes/<id>/tasks.md --direct

# 然后执行
./task_executor.sh
```

**工作量：** 1-2 天。改动集中在 `issue_to_tasks.sh`。
**收益：** 每轮执行节省 2 次 Claude 调用（约 40% token 节省），specs 更精确可追溯。

### 方向 C：loop 的反向写回 OpenSpec

**目标：** 执行完毕后，将结果同步回 OpenSpec。

**执行前——OpenSpec：**

```
openspec/
├── specs/
│   └── todo-app/requirements.md    ← 定义需求
└── changes/
    └── add-todo-crud/
        ├── proposal.md
        ├── design.md
        └── tasks.md                ← 定义任务
```

**执行后——loop 写回：**

```
openspec/
├── specs/
│   └── todo-app/requirements.md    ← 不变（或 delta ADDED）
└── changes/
    └── add-todo-crud/
        ├── proposal.md
        ├── design.md
        ├── tasks.md
        ├── status.md               ← NEW: 各 task 执行状态
        └── verification.md         ← NEW: AC 验证结果 + git commit 引用
```

**`status.md` 内容示例：**

```markdown
# Change Status: add-todo-crud

- **执行时间:** 2026-07-07 10:30
- **分支:** auto/task-run-20260707-103000
- **全部完成:** 4/4 ✅

## Task 状态

| Task | Status | Commit | AC |
|------|--------|--------|----|
| TASK-001: 后端 CRUD API | ✅ done | a1b2c3d | ✅ 通过 |
| TASK-002: 前端列表页 | ✅ done | e4f5g6h | ⏭ 跳过 |
| TASK-003: 排序筛选 | ✅ done | i7j8k9l | ✅ 通过 |
| TASK-004: WebSocket | ✅ done | m0n1o2p | ✅ 通过 |

## 变更文件

- `src/api/tasks.py` — 新增 CRUD 路由
- `src/pages/TodoList.tsx` — 列表页面
- `src/components/SortBar.tsx` — 排序组件
- `src/ws/manager.py` — WebSocket 管理器
```

**工作量：** 1 天。`task_executor.sh` 执行完毕后新增一个写回步骤。
**收益：** OpenSpec 从"规约工具"变成"规约+状态追踪"，实现需求→交付的闭环。

### 方向 D：loop 替代 Superpowers 的 executing-plans

**目标：** 用 `task_executor.sh` 的健壮执行能力，替换 Superpowers 内置的手动 task 执行。

**Superpowers 当前的问题：**
- `executing-plans` 让 Claude Code 在当前上下文逐个运行 task 脚本
- 大项目时 context 窗口膨胀
- 没有重试/断点续跑/状态持久化
- 不能并行

**替换后：**

```markdown
# superpowers/skills/executing-plans/SKILL.md 修改

## 执行阶段

当计划已写好、task 文件已生成时：

1. 检查是否有 task 文件：
   ls tasks/TASK-*.md

2. 如果使用 loop，调用 task_executor 执行：
   cd /path/to/prd-pipeline && ./task_executor.sh

   loop 会处理：
   - 拓扑排序
   - 自动重试（最多 3 次）
   - Git 自动 commit
   - 失败清理工作区
   - 生成 summary.md 报告

3. 读取执行报告给开发者：
   cat 04-execution/summary.md
```

**工作量：** 改 1 个 SKILL.md 文件，约 20 行。
**收益：** Superpowers 从"指导你手动跑"变成"自动执行并汇报"。

### 方向 E：三合一（完整链路）

将 A→D 全部串起来，形成完整的开发工作流：

## 四、核心问题：Agent 支持全景分析

### 4.1 问题本质

```
loop 执行层硬编码:
  echo "$prompt" | claude -p ...     ← 写死了 "claude"

融合后，其他 Agent 用户无法使用:
  codex, cursor, copilot, gemini, opencode ...  ← 各有各的 CLI 调用方式
```

但 loop 的**融合接口不依赖 Agent**：

```
融合接口（方向 B/C）:
  OpenSpec specs → 读文件 → TASK-NNN.md → 执行 → 写回 OpenSpec status.md
                    ↑ 纯文件 I/O        ↑ 可配置 CLI     ↑ 纯文件 I/O
                    不调 Claude         不依赖 Claude    不调 Claude
```

真正需要调 Agent 的只有 `task_executor.sh` + 分解层脚本中那几条管道命令。

### 4.2 2026 年 AI 编码 Agent 全景

#### 主要 CLI Agent 对比

| 特性 | Claude Code | Codex CLI | Copilot CLI | Cursor CLI | Gemini CLI | OpenCode |
|------|------------|-----------|-------------|------------|------------|----------|
| **开发商** | Anthropic | OpenAI | GitHub/MS | Cursor | Google | 开源社区 |
| **开源** | ❌ | ✅ Apache-2.0 | ❌ | ❌ | ✅ Apache-2.0 | ✅ MIT |
| **CLI 管道** | `-p` | `-p` | `-p` | `--prompt-stdin` | 管道模式 | `-p` |
| **SKILL.md** | ✅ 原生 | ✅ 交叉标准 | ✅ 交叉标准 | ✅ 交叉标准 | ✅ 交叉标准 | ⚠️ 有限 |
| **Hook 系统** | ✅ 30+ 事件 | ❌ | ✅ sessionStart | ⚠️ 有限 | ❌ | ❌ |
| **免确认** | --dangerously-skip-permissions | full-auto mode | --allow-tool | 不支持 | --skip-confirm | --yes |
| **无会话** | --no-session-persistence | 每次独立 | 有限制 | 不支持 | --no-history | --no-session |
| **JSON 输出** | --output-format json | --json | 不支持 | 不支持 | --output-format json | --json |
| **Token 效率** | ~4× 基准 | **1× 基准** | ~2× | ~3× | ~2× | 模型依赖 |
| **SWE-bench** | 88.6% | 88.7% | ~75% | ~73% | ~65% | 模型依赖 |
| **人月订阅** | $20 | $20 | $10 | $20 | 免费(1000次/天) | 免费(BYOK) |

#### CLI 管道模式兼容性

| Agent | 管道命令 | 能否直接替换 Claude? |
|-------|---------|---------------------|
| **Claude Code** | `echo "prompt" \| claude -p ...` | ✅ 基准 — 当前默认 |
| **Codex CLI** | `echo "prompt" \| codex -p ...` | ✅ 可替换，但 flag 不同 |
| **Copilot CLI** | `echo "prompt" \| copilot -p ...` | ✅ 可替换，有 `-p` 模式 |
| **Cursor CLI** | `echo "prompt" \| cursor --prompt-stdin ...` | ✅ 可替换，flag 名不同 |
| **Gemini CLI** | `echo "prompt" \| gemini -p ...` | ✅ 可替换 |
| **OpenCode** | `echo "prompt" \| opencode -p ...` | ✅ 可替换 |

**所有主流 AI CLI Agent 都支持管道模式。** 这是 POSIX 标准能力，差异仅在于 flag 名称。

#### 无确认执行能力（关键差异）

自动化循环最关键的 flag — 不弹确认框直接执行：

| Agent | Flag | 是否默认 |
|-------|------|---------|
| Claude Code | `--dangerously-skip-permissions` | 非默认，需显式传入 |
| Codex CLI | full-auto mode | 需配置文件切换 |
| Copilot CLI | `--allow-tool` | 非默认，需显式传入 |
| Cursor CLI | **不支持** | ❌ 只能手动确认每次操作 |
| Gemini CLI | `--skip-confirm` | 非默认，需显式传入 |
| OpenCode | `--yes` | 非默认，需显式传入 |

**Cursor CLI 不支持无确认模式，不适合自动化执行层。** 这个约束直接影响融合方案。

#### 会话独立性（重要）

自动化循环需要每次调用独立、不继承历史：

| Agent | 控制方式 | 是否可行 |
|-------|---------|---------|
| Claude Code | `--no-session-persistence` | ✅ |
| Codex CLI | 每次`-p`调用天然独立 | ✅ |
| Copilot CLI | `-p` 模式有限支持 | ⚠️ 有会话残留风险 |
| Cursor CLI | 无控制方式 | ❌ |
| Gemini CLI | `--no-history` | ✅ |
| OpenCode | `--no-session` | ✅ |

#### Hook/事件系统（Superpowers 依赖）

Superpowers 通过 `session-start` hook 自动注入 context：

| Agent | Hook 机制 | Superpowers 兼容性 |
|-------|----------|-------------------|
| Claude Code | 30+ 生命周期事件，原生 hooks 目录 | ✅ 完整兼容 |
| Codex CLI | 无 Hook 系统 | ❌ 不能自动加载，需手动 /skill |
| Copilot CLI | `sessionStart`/`sessionEnd` hook | ✅ 2026 GA 新增，可移植 |
| Cursor CLI | ⚠️ 有限 `.cursorrules` 注入 | ⚠️ 部分兼容 |
| Gemini CLI | 无 Hook 系统 | ❌ 不能自动加载 |
| OpenCode | 无 Hook 系统 | ❌ 不能自动加载 |

#### SKILL.md 的跨 Agent 生态

2026 年最重要的趋势：**SKILL.md 成为跨 Agent 标准**。Claude Code 原生支持的 Skill 格式，现已被 Cursor、Gemini CLI、Codex CLI、Copilot CLI 共同采用。这意味着：

- 为 Claude Code 写的 SKILL.md，其他 Agent 基本能读
- 但不同 Agent 对 Skill 指令的**执行程度**不同（有些只当提示词，有些严格执行门禁）

### 4.3 Agent 兼容性矩阵（完整版）

将融合后的 pipeline 按 6 个步骤拆分，逐个评估 Agent 兼容性：

```
步骤：
0. 环境安装（一次性的）
1. OpenSpec 规约（Agent 帮助写 specs）
2. loop 拆解（PRD→Issues→Tasks）
3. loop 执行（task_executor.sh）
4. loop 写回 OpenSpec（纯文件操作）
5. Superpowers 流程管控（review/finish）

评分：
✅ = 完整支持
⚠️ = 部分支持，有局限
❌ = 不支持
N/A = 不涉及
```

#### 兼容性矩阵

| Agent | 步骤0 | 步骤1 | 步骤2 | 步骤3 | 步骤4 | 步骤5 | 综合评级 |
|-------|-------|-------|-------|-------|-------|-------|---------|
| **Claude Code** | ✅ | ✅ | ✅ | ✅ | N/A | ✅ | **A+ — 完整** |
| **Codex CLI** | ✅ | ✅ | ⚠️ | ⚠️ | N/A | ❌ | **B — 部分可用** |
| **Copilot CLI** | ✅ | ✅ | ⚠️ | ⚠️ | N/A | ⚠️ | **B — 部分可用** |
| **Gemini CLI** | ✅ | ✅ | ⚠️ | ⚠️ | N/A | ❌ | **C — 基本可用** |
| **OpenCode** | ✅ | ✅ | ⚠️ | ✅ | N/A | ❌ | **C — 基本可用** |
| **Cursor CLI** | ✅ | ✅ | ⚠️ | ❌ | N/A | ⚠️ | **D — 执行层不可用** |

#### 详细限制说明

**Claude Code (A+)：**
- 全部功能可用
- 唯一原生支持 Superpowers hooks 的 Agent
- SKILL.md、hooks、sub-agents、MCP 全部兼容

**Codex CLI (B) —— 最有潜力追赶：**
- 执行层：`echo "prompt" | codex -p ...` ✅ 管道支持，Token 最省
- 执行层：full-auto mode ✅ 可无确认执行
- 执行层：`--json` ✅ 结构化输出
- Superpowers：❌ 没有 hook 系统，不能自动注入流程
- 分解层：SWE-bench 88.7%，与 Claude 相当
- **瓶颈：** 没有 session-start hook，每次需要手动 `/skill-name`
- **缓解：** 可以通过 `codex.md`（项目配置文件）预置指令模拟部分 skill 行为

**Copilot CLI (B)：**
- 执行层：`echo "prompt" | copilot -p ...` ✅ `-p` 模式支持
- 执行层：`--allow-tool` ✅ 可授权工具使用
- Superpowers：✅ 有 `sessionStart`/`sessionEnd` hook！
- **优势：** GP 后新增的 hook 系统使 Superpowers 可以移植
- **瓶颈：** `-p` 模式的会话控制不如 Claude Code 完善，长任务可能中断
- **注意：** Copilot CLI 的 hook 路径是 `.github/hooks/`，不是 `.claude/hooks/`

**Gemini CLI (C)：**
- 执行层：✅ 管道 + 1M 上下文
- 执行层：`--skip-confirm` ✅ 可无确认
- 免费，性价比高
- Superpowers ❌
- SWE-bench 65%，分解层质量明低

**OpenCode (C)：**
- 执行层：✅ 管道 + 免费 BYOK
- 并行多 Agent 执行能力强
- Superpowers ❌
- SWE-bench 依赖后端模型，不固定

**Cursor CLI (D)：**
- 执行层：❌ **不支持无确认模式**，每次操作需人工确认
- 不适合自动化 pipeline 场景
- 适合人工交互式编码，不适合循环执行器

### 4.4 解决方案：Agent 抽象层（升级版）

#### 现状

```bash
# task_executor.sh 当前写法
echo "$prompt" | \
    claude -p --dangerously-skip-permissions --no-session-persistence \
        --output-format text \
        >"$log_file" 2>"$err_file"
```

#### 改造方案一：AGENT_CMD 环境变量（轻量，推荐起步）

```bash
# task_executor.sh
# Agent CLI 配置 — 完整命令字符串，包含所有 flag
# 格式示例:
#   Claude:   claude -p --dangerously-skip-permissions --no-session-persistence --output-format text
#   Codex:    codex -p
#   Copilot:  copilot -p --allow-tool
#   Gemini:   gemini -p --skip-confirm
#   OpenCode: opencode -p --yes
AGENT_CMD="${AGENT_CMD:-claude -p --dangerously-skip-permissions --no-session-persistence --output-format text}"

echo "$prompt" | eval "$AGENT_CMD" > "$log_file" 2>"$err_file"
```

**改动量：** 3 个文件各改 1 行（`task_executor.sh`、`prd_to_spec.sh`、`spec_to_issues.sh`）

**用法示例：**

```bash
# Claude（默认，无需配置）
./task_executor.sh

# Codex CLI
AGENT_CMD="codex -p" ./task_executor.sh

# Copilot CLI
AGENT_CMD="copilot -p --allow-tool" ./task_executor.sh

# Gemini CLI
AGENT_CMD="gemini -p --skip-confirm --no-history" ./task_executor.sh

# OpenCode
AGENT_CMD="opencode -p --yes" ./task_executor.sh
```

#### 改造方案二：结构化 Agent 配置（完整，适合长期）

如果 flag 差异太大，可以用结构化配置：

```bash
# lib.sh 新增
detect_agent_config() {
    AGENT_TYPE="${AGENT_TYPE:-claude}"
    case "$AGENT_TYPE" in
        claude)
            AGENT_BIN="claude"
            AGENT_ARGS="-p --dangerously-skip-permissions --no-session-persistence --output-format text"
            ;;
        codex)
            AGENT_BIN="codex"
            AGENT_ARGS="-p"
            ;;
        copilot)
            AGENT_BIN="copilot"
            AGENT_ARGS="-p --allow-tool"
            ;;
        gemini)
            AGENT_BIN="gemini"
            AGENT_ARGS="-p --skip-confirm --no-history"
            ;;
        opencode)
            AGENT_BIN="opencode"
            AGENT_ARGS="-p --yes"
            ;;
        cursor)
            AGENT_BIN="cursor"
            AGENT_ARGS="--prompt-stdin"
            warn "Cursor CLI 不支持无确认模式。将跳过需要自动执行的步骤。"
            ;;
        *)
            err "不支持的 Agent 类型: $AGENT_TYPE"
            exit 1
            ;;
    esac
}
```

**但这带来一个问题：** 引入了 loop 对 Agent 类型的知识依赖。当前方案二不优先推荐，保持方案一（`AGENT_CMD` 纯字符串）更灵活。

### 4.5 各 Agent 面对融合障碍的缓解策略

| Agent | 障碍 | 缓解策略 |
|-------|------|---------|
| **Codex CLI** | 无 Superpowers hooks | 在 `codex.md` 中写入 skill 指令；用户手动触发 |
| **Copilot CLI** | 会话残留 | `AGENT_CMD` 加 `--no-history` 等价参数；限制每 task 执行时间 |
| **Cursor CLI** | 不能无确认执行 | 改用交互模式 + 人工确认——牺牲自动化换取可用性 |
| **Gemini CLI** | 分解质量低 | 跳过分解层，直接用 OpenSpec 的结构化 specs 作为输入 |
| **OpenCode** | 无 Superpowers hooks | 用 `OPENCODE.md` 预设行为 |

### 4.6 推荐策略

```
默认: Claude Code（A+ 完整支持）
  └─ 如果用户已在使用其他 Agent:
      ├─ Codex CLI → 执行层可用，Superpowers 层需手动触发 skill
      ├─ Copilot CLI → 执行层+部分Superpowers可行，hook路径需移植
      ├─ Cursor CLI → ❌ 不建议用于 pipeline 执行层
      └─ Gemini/OpenCode → 执行层可用，缺少流程管控
```

**最务实的融合策略：**

1. **方案一 `AGENT_CMD` 作为接入点** — 让 codex/copilot/gemini/opencode 用户也能跑执行层
2. **SKILL.md 作为跨 Agent 文档** — 即使没有 hooks，用户也能手动 `/skill-name` 获取指令
3. **Claude Code 作为 Superpowers 层的最佳运行时** — hooks 系统和 `--dangerously-skip-permissions` 是 pipeline 自动化的关键

### 5.2 Agent 无关层 vs Agent 敏感层

```
Agent 无关层（纯文件 I/O，shell 脚本，git）:
  ┌──────────────────────────────────────────────────┐
  │ ● OpenSpec specs/ 读写                            │
  │ ● OpenSpec changes/ status.md 写回               │
  │ ● loop 的 task_board.md / run-manifest.json 生成  │
  │ ● loop 的 summary.json / summary.md 生成          │
  │ ● loop 的 Git 分支管理 / commit / diff 跟踪        │
  │ ● loop 的重试逻辑 / 拓扑排序 / 断点续跑            │
  │ ● 所有进度显示（progress.sh）                      │
  └──────────────────────────────────────────────────┘

Agent 敏感层（依赖 CLI + 模型能力）:
  ┌──────────────────────────────────────────────────┐
  │ ● loop 执行层调 Agent 实现代码（AGENT_CMD 抽象）   │
  │ ● loop 分解层调 Agent 做规划（AGENT_CMD 抽象）     │
  │ ● Superpowers hooks 自动加载流程                   │
  │ ● Superpowers skill 执行质量                       │
  └──────────────────────────────────────────────────┘
```

**融合的核心原则：Agent 无关层不设限，Agent 敏感层用 `AGENT_CMD` 做接入网关。** 这样即便某个 Agent 在特定层有局限，不影响整个 pipeline 的 70% 功能。


### 5.4 升级后的融合价值

```
融合后，一个 PRD→代码 的全流程在不同 Agent 上的体验:

Claude Code 用户:
  1. 说需求 → OpenSpec 自动生成 specs
  2. loop 自动拆解+执行
  3. Superpowers 自动 review+finish
  → 全程语音不离开终端，零人工介入

Codex CLI 用户:
  1. 说需求 → OpenSpec specs
  2. loop 自动拆解+执行（AGENT_CMD=codex）
  3. Superpowers 手动 /requesting-code-review
  → 执行层全自动，流程层需手动触发

Cursor 用户:
  1. 说需求 → OpenSpec specs
  2. loop 拆解（手动确认每一步，因为 Cursor 不支持无确认）
  3. 手动执行 task
  → 不适合全自动化，适合半自动辅助
```

### 5.5 核心结论

| 结论 | 说明 |
|------|------|
| **管道模式是通用的** | 所有主流 AI CLI 都支持 `echo prompt \| cmd -p`，差异仅在于 flag 名 |
| **`AGENT_CMD` 一行改动就够** | 3 个脚本各改 1 行，兼容所有 Agent 的执行层需求 |
| **Cursor 不适合自动化** | 缺乏无确认执行能力，不适合作为 pipeline 的执行 Agent |
| **Superpowers 是 Claude Code 差异化的关键** | hooks 系统目前只有 Claude Code 和 Copilot CLI 支持 |
| **SKILL.md 成为跨 Agent 标准** | 写一次 SKILL.md，多数 Agent 都能读取 |
| **融合后 70% 功能与 Agent 无关** | 报告、Git、进度显示、文件操作——不受 Agent 选择影响 |

## 六、风险与注意事项

1. **OpenSpec 是 Node.js/TypeScript 项目**，loop 是纯 bash。`--spec-dir` 接口要基于文件系统（读 MD 和 JSON），不要引入 Node.js 依赖。

2. **Superpowers 版本 6.x**，接口可能变化。SKILL.md 的 frontmatter 格式需对齐最新版本。当前用的是 `name` + `description`。

3. **执行顺序依赖**：如果同时修改 OpenSpec 的 specs 和 loop 的 task 文件，要注意写锁或串行化。当前 loop 的串行执行天然避免了这个问题。

4. **用户选择权**：融合后所有原始功能应保持可用。不加 `--spec-dir` 时 loop 行为零变化。
