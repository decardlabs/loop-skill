## 第二部分：融合方案的功能完整性评估

### 2.1 OpenSpec → loop 接口：覆盖了哪些？

| OpenSpec 能力 | 融合方案是否覆盖 | 覆盖方式 | 遗漏？ |
|--------------|----------------|---------|--------|
| Proposal 生成 | ✅ 方向 B — loop 读取 specs/ | 文件 I/O | — |
| Delta specs (ADDED/MODIFIED/REMOVED) | ⚠️ 方向 C — 写回 status.md，但不理解 delta | loop 只写状态，不写 spec delta | **遗漏：未实现 delta 写回** |
| `tasks.md` 直接转 task | ✅ 方向 B — 读取 tasks.md 为 task 文件 | `--spec-dir` 模式 | — |
| 验证 (validate) | ❌ 未涉及 | — | **遗漏：loop 不调用 OpenSpec validate** |
| Archive | ❌ 未涉及 | — | **遗漏：执行完后不自动 archive** |
| Stores | ❌ 未涉及 | — | **合理：Stores 是 OpenSpec 内部功能** |
| 自定义 Schema | ❌ 未涉及 | — | **合理：高级功能，可以后续** |
| 30 工具适配 | ❌ 不直接使用 | loop 只关心 specs/ 目录 | **合理：文件格式是通用的** |
| JSON Agent 合约 | ❌ 未涉及 | — | **遗漏：loop 不应该依赖 OpenSpec 的 Node.js CLI，但可以读 JSON** |

**评估：** 方向 B/C 覆盖了 OpenSpec 的核心数据流（specs/tasks + status 写回），但忽略了 3 个重要接口：
1. delta spec 的写回（loop 执行完应该能生成 delta，不只是标记状态）
2. archive 触发器（执行完自动触发 archive）
3. validate 前置检查（执行前验证 specs 完整性）

### 2.2 Superpowers → loop 接口：覆盖了哪些？

| Superpowers 能力 | 融合方案是否覆盖 | 覆盖方式 | 遗漏？ |
|-----------------|----------------|---------|--------|
| Brainstorming | ❌ 方案未涉及 | — | **合理：loop 不替代设计讨论** |
| Writing-plans | ✅ 方向 D — loop 替代 executing-plans | loop 的 task 文件 = Superpowers 的 plan | — |
| SDD (子代理) | ⚠️ 未深度匹配 | loop 是串行执行，非子代理模式 | **合理：模式不同，loop 是 shell 自动化** |
| **executing-plans** | ✅ 方向 D — 直接替换 | `task_executor.sh` 替代手动执行 | — |
| **TDD** | ❌ 方案未涉及 | — | **重要遗漏：loop 执行时不强制 TDD** |
| **Code Review** | ❌ 方案未涉及 | — | **重要遗漏：loop 执行后不自动触发 review** |
| **Verification** | ⚠️ 方向 E — AC 验证 | `--verify-ac` 提供了轻量版本 | AC 验证不等同于 Superpowers 的验证门禁 |
| **Git worktree** | ⚠️ 方向 F — 临时分支 | `init_git_branch()` 类似但不等同 | 临时分支 vs worktree 隔离不同 |
| Finishing branch | ❌ 方案未涉及 | — | loop 的 `generate_summary` 提供合并建议但不自动执行 |
| **Hard gates** | ❌ 方案未涉及 | — | **重要遗漏：loop 自动化绕过了 Superpowers 的质量门禁** |

**评估：** 方向 D 正确定位了 `executing-plans` 作为最合适的替换点，但漏了 3 个关键问题：
1. **TDD 强制** — loop 执行时不会先写测试再写代码
2. **Review 门禁** — loop 执行后不会自动审查代码质量
3. **Hard gates bypass** — 自动化执行天然绕过了 Superpowers 的"先设计再实现"门禁

### 2.3 技术栈差距

| 维度 | OpenSpec | Superpowers | loop | 融合障碍 |
|------|---------|-------------|------|---------|
| 语言 | Node.js/TypeScript | Bash + Markdown | Bash | OpenSpec 的 CLI 无法被 loop 直接调用（除非假设用户已安装） |
| 依赖 | Node.js ≥20.19, npm | 零依赖 | bash 3.2+ | `openspec validate` 需要 Node.js 运行时 |
| 运行方式 | CLI + 斜杠命令 | Hook 注入 + SKILL.md | Shell 脚本 | 融合接口只能是**文件系统** |
| Git 耦合 | 不绑 git | 深度依赖 git | 深度依赖 git | 一致 ✅ |
| 错误契约 | JSON + 退出码 | 无（SKILL.md 是文本指令） | 退出码 + 日志 | **不一致：OpenSpec 用 JSON 结构体，loop 用纯文本日志** |

## 第三部分：有效性评估

### 3.1 每个融合方向的实际价值

| 方向 | 价值 | 风险 | 净有效性 |
|------|------|------|---------|
| **A: loop → Superpowers Skill** | 低 — 只是把 bash 命令包装成 SKILL.md 文本 | 低 — 纯文档变更 | **中等** — 有价值但浅 |
| **B: loop 消费 OpenSpec specs** | 高 — 跳过 2 次 Claude 调用，spec 更精确 | 中 — 需 Node.js 环境 | **高** — 最能体现融合价值 |
| **C: loop 写回 OpenSpec** | 中 — 闭环追溯 | 中 — delta spec 格式复杂 | **中低** — 收益小于实现成本 |
| **D: loop 替代 executing-plans** | 高 — 自动化执行替换手动操作 | 低 — 方向正确 | **高** — 最自然的切入点 |
| **E: 三合一全链路** | 高 — 端到端 | 高 — 依赖前 4 个方向全部实现 | **低** — 目前条件不成熟 |

### 3.2 关键缺口分析

```
融合方案的 5 个方向覆盖了：
  ├─ 数据流（文件 I/O）        ✅ 充分覆盖
  ├─ 执行自动化                ✅ 充分覆盖
  ├─ 报告与追溯                ✅ 充分覆盖
  ├─ 质量门禁（TDD/Review）    ❌ 未覆盖
  ├─ OpenSpec delta 写回       ❌ 未覆盖
  ├─ Superpowers hooks 集成    ⚠️ 部分覆盖
  └─ Agent 抽象               ✅ 方案设计完成但未实施
```

**最大缺口：质量门禁的丧失。**

当前方案让 `task_executor.sh` 全自动执行 task，这意味着：

```bash
# Superpowers workflow 中:
brainstorming (有门禁) → writing-plans (有门禁) → executing-plans (手动执行，有门禁)
                                                          ↓
# 融合后:
brainstorming (有门禁) → writing-plans (有门禁) → task_executor.sh (无人值守，TDD? review? 全跳过)
                                                          ↓
                                                     ❌ TDD 被绕过
                                                     ❌ code review 被绕过
                                                     ❌ verification 被绕过（AC 验证是轻量版）
```

### 3.3 Agent 抽象的验证

`AGENT_CMD` 方案在设计上合理，但有一个实际验证未完成：

| 验证项 | 状态 | 说明 |
|--------|------|------|
| `echo prompt \| claude -p` 工作 | ✅ 已实现 | loop 当前行为 |
| `echo prompt \| codex -p` 工作 | ⚠️ 待验证 | 需要 Codex 实际测试 |
| `echo prompt \| copilot -p` 工作 | ⚠️ 待验证 | Copilot 的 `-p` 模式有限制 |
| `echo prompt \| cursor --prompt-stdin` 工作 | ❌ 已知问题 | Cursor 不支持无确认 |
| 退出码一致性 | ⚠️ 待验证 | 各 Agent 的退出码含义可能不同 |
| 超时行为 | ⚠️ 待验证 | `task_executor` 没有超时控制 |

## 第四部分：合理性评估

### 4.1 架构合理性

```
融合后三层架构：
  规约层 (OpenSpec) —————— Node.js 生态
  执行层 (loop)      —————— Bash 生态
  流程层 (Superpowers) ——— Markdown + Hooks
```

**问题：三种技术栈共存是否合理？**

| 论证 | 结论 |
|------|------|
| 技术栈隔离通过文件系统 | ✅ 合理 — specs 是 MD 文件，不跨语言调用 |
| OpenSpec CLI 不被 loop 直接调用 | ✅ 合理 — 读 MD 不需要 Node.js |
| loop 不依赖 Superpowers hooks | ✅ 合理 — hooks 是 Superpowers 的内部机制 |
| 用户需要安装 3 个项目 | ⚠️ 但这是融合的前提，不算不合理 |

**评估：架构合理。** 文件系统的通用性成功隔离了三种技术栈。OpenSpec 的 Node.js 生态和 loop 的 Bash 生态通过 `.md` 和 `.json` 文件交换数据，没有直接耦合。

### 4.2 依赖合理性

```
融合后用户需要安装：
  npm install -g @fission-ai/openspec    ← 新增
  git clone loop                          ← 已有
  git clone superpowers / plugin install   ← 新增
  claude CLI 或 其他 Agent CLI            ← 已有
```

约 4 个前置依赖。对于"一个命令跑全流程"的目标来说偏重。但这是三项目融合的固有成本，每个项目独立安装原本就需要。

**缓解：** `.agentrc` 配置可以逐步引导用户安装缺失项。

### 4.3 冲突分析

| 冲突点 | loop 的做法 | OpenSpec 的做法 | 是否冲突 |
|--------|------------|----------------|---------|
| Task 定义 | YAML frontmatter + Markdown prompt | `tasks.md` 复选框列表 | **不冲突** — loop 可以读取 tasks.md 并转换为内部格式 |
| 状态管理 | `status: pending/running/done/failed` | `openspec list --json` 的 `status` 字段 | **不冲突** — 可以双向映射 |
| 分支策略 | `auto/task-run-<ts>` 临时分支 | 不涉及 git | **不冲突** — OpenSpec 不关心 git |
| Spec 格式 | 自由 Markdown | 结构化 Given/When/Then | **不冲突** — loop 读任何 MD 文件 |
| 执行方式 | `echo prompt \| AGENT_CMD` | `/opsx:apply` AI chat 内执行 | **不冲突** — 不同的实现策略，可以共存 |

**评估：无结构性冲突。** 三个项目的领域完全正交，融合后没有重叠导致的冲突。

## 第六部分：建议的优先级调整

### 立即实施（Phase 1 — 当前 v0.1.0 之后）

```
1. AGENT_CMD 实施（3 个脚本各改 1 行）
   收益: 为多 Agent 支持打好基础
   风险: 低

2. 方向 A: loop → Superpowers Skill
   收益: Claude Code 内斜杠命令调用
   风险: 低

3. 方向 B: loop 消费 OpenSpec specs（最小版本）
   收益: 节省 token
   风险: 中 — 需要安装 OpenSpec，但不阻塞
```

### 短期实施（Phase 2 — 下一版本）

```
4. 方向 D: loop 替代 executing-plans
   修改 Superpowers 的 executing-plans/SKILL.md
   收益: 自动化执行，免手动操作
   风险: 低

5. 质量门禁补偿
   在 task_executor.sh 执行每个 task 后，增加:
   - 自动运行测试（如果有）
   - 调用 requesting-code-review（如果可用）
   收益: 弥补融合方案的最大缺口
   风险: 中 — 需要协调 Superpowers 的 review 流程
```

### 中期实施（Phase 3）

```
6. 方向 C: OpenSpec delta 写回
   收益: 闭环追溯
   风险: 中 — delta spec 格式解析复杂

7. OpenSpec validate 前置集成
   在执行前自动运行 openspec validate
   收益: 提前发现 specs 问题
   风险: 低 — 只是调用 CLI

8. 方向 E: 三合一（条件成熟后）
   依赖: Phase 1-3 全部完成
```

## 结论

**融合方案总体方向正确，但有两个必须处理的缺口：**

1. **质量门禁丧失（🔴 高优先级）** — loop 的全自动执行必然绕过 Superpowers 的 TDD/review/verification 门禁。需要设计"执行后门禁补偿"机制，比如每个 task 执行完后自动触发 `requesting-code-review` 和测试运行。

2. **OpenSpec delta 写回缺失（🟡 中优先级）** — 只写 status.md 不写 delta spec，使得 OpenSpec 的 archive 流程无法闭环。但这不是阻塞性的——用户可以手动 archive。

**建议的起步顺序：AGENT_CMD → 方向 A → 方向 B(最小) → 方向 D → 质量门禁补偿**
