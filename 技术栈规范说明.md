## 二、前端技术栈（frontend/）

### 语言与框架

| 项目 | 选择 | 说明 |
|---|---|---|
| 语言 | **TypeScript** | 必须开启 `strict` 模式 |
| 框架 | **React 18+** | 函数组件 + Hooks，禁止 class 组件 |
| 构建工具 | **Vite** | 替代 Webpack，启动更快 |
| 路由 | **React Router v6** | 声明式路由 |
| 状态管理 | **Zustand**（轻量）或 **Redux Toolkit**（复杂） | 优先 Zustand |
| 数据请求 | **TanStack Query (React Query)** | 负责缓存、重试、轮询 |
| UI 组件库 | **Ant Design** 或 **MUI** | 根据项目风格选择 |
| CSS 方案 | **Tailwind CSS** | 原子化 CSS，AI 生成效率高 |

### TypeScript 编译配置（tsconfig.json 关键项）

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "lib": ["DOM", "DOM.Iterable", "ES2022"],
    "strict": true,              // 必须开启
    "noImplicitAny": true,       // 禁止隐式 any
    "strictNullChecks": true,    // 严格 null 检查
    "noUnusedLocals": true,      // 禁止未使用变量
    "noUnusedParameters": true,  // 禁止未使用参数
    "noImplicitReturns": true,   // 函数必须有返回值
    "skipLibCheck": true,
    "esModuleInterop": true,
    "forceConsistentCasingInFileNames": true,
    "moduleResolution": "bundler",
    "resolveJsonModule": true,
    "isolatedModules": true,
    "jsx": "react-jsx"
  }
}
```

### 代码规范

- **ESLint**：使用 `eslint-config-airbnb` + `@typescript-eslint` 插件
- **Prettier**：统一代码格式，AI 生成后自动格式化
- **Husky + lint-staged**：提交前自动 lint 和格式化
- 所有组件必须使用 **函数式组件 + TypeScript 接口定义 props**
- 所有 API 响应必须定义 **Response 类型**，禁止使用 `any`

### 构建与部署

- 开发：`npm run dev`（Vite 热更新）
- 构建：`npm run build` → 输出到 `dist/` 目录
- 产物：静态文件（HTML/CSS/JS），由后端 Go 或 Nginx 托管

## 三、后端技术栈（backend/）

### 语言与框架

| 项目 | 选择 | 说明 |
|---|---|---|
| 语言 | **Go 1.22+** | 使用最新稳定版 |
| Web 框架 | **Gin** | 轻量、高性能、AI 生成质量高 |
| 依赖注入 | **Wire**（Google） | 编译时 DI，零运行时反射开销 |
| ORM | **GORM v2** | 功能全面，AI 熟悉度高 |
| 配置管理 | **Viper** | 支持 JSON/YAML/环境变量 |
| 日志 | **Zap**（Uber） | 高性能结构化日志 |
| 验证 | **go-playground/validator** | 结构体 tag 验证 |
| API 文档 | **Swaggo** | 从注释生成 Swagger 文档 |

### 项目结构（标准 Go 布局）

```
backend/
├── cmd/
│   └── server/           # 主入口
│       └── main.go
├── internal/             # 私有业务代码
│   ├── handler/          # HTTP 处理器
│   ├── service/          # 业务逻辑层
│   ├── repository/       # 数据访问层
│   ├── model/            # 数据模型（GORM 结构体）
│   ├── middleware/       # 中间件
│   └── config/           # 配置结构体
├── pkg/                  # 可复用的公共库
├── api/                  # API 契约（OpenAPI/Swagger）
├── scripts/              # 数据库迁移等脚本
├── go.mod
└── go.sum
```

### Go 编译模式

```bash
# 开发模式（带调试信息）
go build -o bin/server cmd/server/main.go

# 生产模式（优化体积和性能）
CGO_ENABLED=0 GOOS=linux GOARCH=amd64 \
  go build -ldflags "-s -w" -o bin/server cmd/server/main.go

# 交叉编译（Mac 编译 Linux 产物）
CGO_ENABLED=0 GOOS=linux GOARCH=arm64 \
  go build -o bin/server cmd/server/main.go
```

**关键编译标志说明：**
- `CGO_ENABLED=0`：禁用 CGO，生成**纯静态二进制**，容器部署无需额外依赖
- `-ldflags "-s -w"`：去掉调试符号，减小体积（生产环境用）
- 产物是**单文件可执行程序**，无需安装运行时

### 代码规范

- 使用 `gofmt` 统一格式（AI 生成后必须执行）
- 使用 `go vet` 检查常见错误
- 使用 `staticcheck` 做静态分析
- 所有导出的结构体字段必须有 **JSON tag**
- 所有错误必须显式处理（`if err != nil`），禁止忽略
- 禁止使用 `panic` 处理业务错误，只用于不可恢复的致命错误
- 使用 `context.Context` 传递请求上下文（超时、取消、trace）

### 运行与部署

- 开发：`go run cmd/server/main.go`
- 构建：`go build -o bin/server cmd/server/main.go`
- 运行：`./bin/server`
- 容器化：基于 `scratch` 或 `alpine` 镜像，产物仅几 MB 到几十 MB

## 四、前后端接口契约

### 通信协议

- **HTTP + JSON**：标准 RESTful API
- 所有接口返回统一格式：

```json
{
  "code": 0,
  "message": "success",
  "data": { ... }
}
```

### 类型共享方案

- 在 `docs/api-types/` 目录下，用 **TypeScript 接口** 定义所有 API 的请求/响应类型
- AI 生成后端 Go 代码时，参考这些类型定义生成对应的 Go struct
- 确保前后端字段名、类型一致（使用 `json` tag 映射）

示例：

```typescript
// docs/api-types/user.ts
export interface User {
  id: number;
  username: string;
  email: string;
  createdAt: string; // ISO8601
}
```

```go
// backend/internal/model/user.go
type User struct {
    ID        uint      `json:"id" gorm:"primaryKey"`
    Username  string    `json:"username"`
    Email     string    `json:"email"`
    CreatedAt time.Time `json:"createdAt"`
}
```

> 你可以把下面这段直接放进 AI 的系统提示词中：

```
你是一个全栈开发 AI，遵守以下技术约束：

【前端】
- 语言：TypeScript，必须使用严格类型，禁止 any
- 框架：React 函数组件 + Hooks
- 所有 props、state、API 响应必须定义 interface/type
- 使用 TanStack Query 管理服务端状态
- 使用 Tailwind CSS 写样式

【后端】
- 语言：Go 1.22+，使用标准库 + Gin 框架
- 所有结构体必须有 json tag
- 所有错误必须 if err != nil 显式处理
- 使用 context.Context 传递请求上下文
- 使用 GORM 操作数据库

【通用】
- 每写一个功能模块，必须同步生成对应的单元测试
- 所有 API 必须符合统一的 JSON 响应格式 {code, message, data}
- 代码风格：前端遵循 ESLint + Prettier，后端遵循 gofmt
- 禁止写注释中的 TODO，所有功能必须完整实现
```

## 📌 总结

| 维度 | 前端 | 后端 |
|---|---|---|
| 语言 | TypeScript（strict 模式） | Go 1.22+ |
| 框架 | React + Vite | Gin |
| 类型安全 | TS 编译时检查 | Go 编译时检查 |
| 构建产物 | 静态文件（dist/） | 单文件二进制 |
| 部署方式 | Nginx / Go 静态托管 | 直接运行二进制 / Docker |
| AI 友好度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |

这套组合的优势在于：**两端都有强类型编译检查**，AI 生成的代码在编译阶段就能暴露大部分 bug，运行时错误率极低；同时 Go 的单文件编译和 TS 的零运行时开销，保证了最终产物的运行效率。
