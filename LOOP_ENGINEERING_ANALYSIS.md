## 一、核心定义

> **Loop Engineering = 用系统替代你自己作为 prompt agent 的那个人**

由 Google Chrome 工程师 **Addy Osmani** 正式命名。灵感来源于同一周内三个独立的声音指向同一件事：

| 人物 | 来源 | 原话 |
|------|------|------|
| **Peter Steinberger** (OpenClaw 作者) | 博客 (800万+ 阅读) | "你不应该再 prompt 编码 agent 了，你应该设计 loop 来 prompt 你的 agent" |
| **Boris Cherny** (Anthropic, Claude Code 负责人) | 演讲 | "我不再 prompt Claude 了。我有一堆 loop 在跑，它们来 prompt Claude 并决定做什么" |
| **Addy Osmani** (Google Chrome) | 博客 / Substack | 正式命名为 **Loop Engineering** |

### 一句话总结范式转变

**旧世界**：你坐在那里逐行给 agent 下指令。它做完一件事就停下来等你下一行。你是 loop 内部的人类时钟，每一个 tick 都必须来自你。

**新世界**：你设计一个自己能 tick 的东西，一个 tick 接着一个 tick。它由定时器驱动，生出小帮手去做事，把自己的结果反馈给自己。你不再在 loop 内部——你在它外面，看着它转动。

> "你的价值曾经在于'知道如何指挥'。现在指挥被你自己写的 loop 取代了。你的价值转移到'知道如何构建 loop，以及在 loop 内安装一个能说不的检查'。"

## 二、四层技术栈

这一系列 "XX Engineering" 不是相互替代的——它们**堆叠**，每一层关注更大的问题：

| 层次 | 关注点 | 核心问题 | 失效影响 |
|------|--------|---------|---------|
| **Prompt Engineering** | 写一句好 prompt | 该跟模型说什么 | 当场可见，改一行就行 |
| **Context Engineering** | 窗口里该放什么 | 该检索/总结/清除什么 | 模型答偏，能察觉到 |
| **Harness Engineering** | 武装单次运行 | 给什么工具、允许什么动作、怎样算完成 | 运行出问题可调试 |
| **Loop Engineering** | 让它自己反复运行 | 如何让它自己 tick 自己 | **你睡着时出错，几天后才知** |

> "Loop engineering 坐在 harness 上面一层。下面那层武装单次运行，上面这层让它自己反复运行。"

**为什么非要分这么细？** 因为每一层出问题的方式不同，安装检查的位置也不同。Prompt 出问题当场就能发现；但 Loop 层在你睡觉时运行，改了你没看过的代码，把错误喂给下一轮——你可能几天后才发现。

> 层次越高，你离现场越远，错误堆积的时间越长。这正是 loop engineering 的真正难点从来不是构建 loop，而是**在它内部安装一个能阻止它的东西**。

## 三、一个 Loop 的五步运动 (The Five Moves)

听 "loop" 这个词，很容易联想到一段代码无意义地空转。但 loop engineering 中的 loop 不是空转——**每一圈都做具体的事**。

### 运动总览

```
1. 发现 (Discovery)  ── 自动找到"这一轮该做什么"
      │ 技能读取 CI 失败 / 新 Issue / 最近提交
      ▼
2. 移交 (Handoff)    ── 把任务交给 agent，隔离执行
      │ 每个发现开一个独立的 git worktree
      ▼
3. 验证 (Verification) ── 换另一个 agent 来挑刺、"说不"
      │ 不同指令、不同模型的 sub-agent 进行 review
      ▼
4. 持久化 (Persistence) ── 把结果写进磁盘 / PR / 看板
      │ "agent 会忘，repo 不会"
      ▼
5. 调度 (Scheduling)  ── 自动化触发，让它明天继续转
      │ 没有调度，前 4 步只是一次手动运行
```

### 每步详解

**Move 1: 发现 (Discovery)**

一个 loop 在每一轮做的第一件事是搞清楚"这一轮该做什么"。关键：**让 agent 自己找到工作**，而不是你每天早上手动列任务。

> "触发一个 `$skill-name`，而不是把一大段指令贴进 cron——贴进去就没人会再更新了。"

发现的质量设定了整个 loop 的上限。如果发现的工作没有价值，其他四步做得再好也只是在**精细地做无用功**。

**Move 2: 移交 (Handoff)**

把任务从调度系统移交到干活 agent 的手中。**每个发现开一个独立的 worktree**——git 机制让多个 agent 在独立的工作目录里改代码，互不干扰。

> "两个 agent 同时改同一个文件 = 两个工程师 commit 到同一行。"

移交的设计诀窍：不是把整份工作丢给一个 agent，而是切成隔离的小任务再分发。任务边界越清晰，验证和合并越容易。

**Move 3: 验证 (Verification)**

整个 loop 最容易偷工减料、也最不能省的一步。**用不同的 agent 做 reviewer**，指令不同，有时模型也不同。

> "写代码的 agent 给自己打分太宽松了。换一个 agent，专门挑刺，它能抓住第一个 agent 说服自己放过去的问题。"

> "Loop 最难的部分不是 loop 本身，而是在它内部放一个能说不的东西。没有真正检查的 loop，不过是一个 agent 在反复点头认可自己。"

**Move 4: 持久化 (Persistence)**

验证通过的结果需要落到**能跨对话存活**的地方——代码仓库、Issue 系统、状态文件。

> "Agent 会忘记，repo 不会。Memory 必须在磁盘上，不能在上下文里。"

没有持久化，每一轮 loop 都像第一次睁开眼睛，永远从零开始。

**Move 5: 调度 (Scheduling)**

前四步加起来只是"一轮"。**调度把'我手动跑了一次'变成'它自己持续转'**

> "自动化才是让 loop 成为真正的 loop 的东西，而不是你碰巧跑了很多次的一次性任务。"

### 五步速查表

| Move | 做什么 | 在 Triage Loop 中的体现 |
|------|--------|----------------------|
| 发现 | 自己找到这一轮的工作 | skill 读取 CI 失败 / Issues / 提交 |
| 移交 | 把任务移交给 agent，隔离 | 每个发现开一个 worktree |
| 验证 | 换一个 agent 来说不 | 第二个 sub-agent 对照测试做 review |
| 持久化 | 把状态写到对话之外 | 自动开 PR + 更新 Inbox + 写状态文件 |
| 调度 | 让它一圈接一圈地自动转 | 早晨自动化自己启动 |

## 四、构建 Loop 的六大部件 (Six Parts)

"五步运动"描述"一轮里发生什么"；"六大部件"描述"你需要手里有什么才能让它转起来"。

| 部件 | 本质 | 对应哪步运动 | 一句话 |
|------|------|-------------|--------|
| **Automations** | 定时器 / 触发器 | Scheduling | "让 loop 成为真正的 loop" |
| **Worktrees** | Git 隔离工作目录 | Handoff | "两个 agent 同时改同个文件 = 两个工程师 commit 到同一行" |
| **Skills** | SKILL.md 固化项目知识 | Discovery | "触发 `$skill-name`，而不是贴一墙指令" |
| **Connectors** | MCP 连接外部系统（Issue 追踪器、DB、Slack） | Persistence / Discovery | "只能看到文件系统的 loop 是很小的 loop" |
| **Sub-agents** | 生成者和评判者分离 | Verification | "写代码的 agent 给自己打分太宽松了" |
| **Memory** | 磁盘上的持久状态 | Persistence | "agent 会忘，repo 不会" |

### 各部件详解

**Part 1: Automations (自动化 / 调度)**

loop 的引擎。挂在定时器或触发器上，时间到了自动发现和处理。

- 触发的东西应该是一个 **skill**（`$skill-name`），而不是一大段贴在 cron 里的 prompt。贴进去就没人会更新了。
- 调度形式选择：
  - 本地（`/loop`）：需要机器开机
  - 云端（Cloud Routines / GitHub Actions）：机器关掉也能跑

**Part 2: Worktrees (Git 隔离)**

一种 git 机制，在同一个仓库里开多个相互独立的工作目录。每个 agent 在自己的 worktree 里改代码。

- 价值随并行度放大：单 agent 运行无所谓，但五个 agent 同时改五个 bug 时，不隔离就是一团乱麻。
- "Cattle not pets"——每个 agent 的工作环境是可替换的牲口，不是需要呵护的宠物。坏了一个就新开一个，不去修。

**Part 3: Skills (固化知识)**

把项目知识永久固化在单个文件（SKILL.md）中，agent 每次运行都能使用，不需要每次都重新推导。

- 解决 **intent debt（意图债务）**：每次都要重新告诉 agent"这个项目是什么、规则是什么、坑在哪里"。
- 发现步骤是 triage skill 而不是裸 prompt，正是这个道理——skill 可复用可维护，prompt 不行。

**Part 4: Plugins & Connectors (MCP 连接器)**

loop 连接外部世界的接口——Issue 追踪器、数据库、Slack、PR 系统。技术基础是 **MCP (Model Context Protocol)**。

- 连接器决定了 loop 的"视野半径"：只看到文件系统就只看到几个目录；连上 Issue 追踪器就知道还有哪些工作没做；连上 Slack 就能通知你结果。
- MCP 的跨工具兼容性：在 Codex 写的连接器通常可以直接用在 Claude Code 上。

**Part 5: Sub-agents (生成者和评判者分离)**

把"写代码的人"和"评判代码的人"拆成两个 agent。

> 直觉上你可能认为要让 agent 学会批评自己，需要花功夫训练生成器。但实践正好相反：**调出一个独立的、足够挑剔的评判者，比让生成器对自己的工作严格要容易得多**。

**Part 6: Memory (持久状态)**

loop 的"记忆"——一份 markdown 文件或一张 Linear board。

- Memory 和 Context 的区别：
  - **Context** = agent 这一轮能看到的东西，窗口刷新就清空
  - **Memory** = 写到磁盘上的东西，跨轮、跨天、跨整个 loop 的生命周期

## 五、核心洞察：生成器与评估器分离 (§05)

**这是全书最关键的一章。**

### 问题：Agent 总给自己的代码打高分

> "当你让 agent 评估自己的工作结果时，它倾向于自信地赞美——即使一个人类能明显看出质量平庸。"

这不是智能问题，这是**结构性问题**。agent 写代码的上下文里已经装满了它那样写的理由。当它看自己的代码时，看到的不是结果——它看到的是说服自己的整个推理链条。

人类也一样：你写点东西读回去，每一遍都觉得更顺——不是因为真的写得好，而是因为脑子里已经有一个"它应该是什么样"的版本。

### 解决方案：从 GAN 借来的 Generator-Evaluator 结构

Anthropic 工程师 **Prithvi Rajasekaran** 借用了 **生成对抗网络 (GAN)** 的思路：

- **Generator（生成器）**：写代码
- **Evaluator（评估器）**：挑错，**必须是不同的 agent**

> "调出一个独立的、足够怀疑的评估器，比让生成器学会批评自己的工作要容易得多。"

### 评估器的设计原则

1. **必须是不同的 agent**——不同的指令集，有时不同的模型
2. **默认立场应该是怀疑**——社区实践中，评估器的常用设定是"这段代码在没有被证明正确之前都是坏的"
3. **应该动手操作验证**——不仅仅是读代码，而是用 Playwright MCP 打开页面、点击按钮、截屏、检查 DOM

### 产品化：`/goal`

Claude Code 的 `/goal` 命令把这个结构产品化了：

```
/goal all tests in test/auth pass and the lint step is clean
```

关键：**谁来判断条件是否满足？** 不是干活的 agent——每次轮转后，一个小而快的模型独立检查条件。它没参与这一轮的工作，不带有自我说服的包袱。

| 模式 | 谁决定"完成" | 问题/优势 |
|------|-------------|---------|
| Agent 自我评分 | 干活的 agent | 带有自我说服的包袱，倾向于赞美自己 |
| `/goal` | 每次轮转后换一个新模型 | 无包袱，只能看客观条件 |

> "Maker-checker 不是 AI 发明的。银行转一大笔钱时，录入的人和审核的人必须是两个不同的人。这个规则存在了几十年，原因和这里一样。"

## 六、三个真实案例 (§06)

### 1. Addy 的早间 Triage Loop

一个人、一台机器，每天早晨自动运行：

- 自动化在黎明启动
- Triage skill 读取 CI 失败、未关 Issue、最近提交
- 每个值得处理的发现开一个 worktree
- 一个 sub-agent 起草 fix，第二个 sub-agent 做 review
- 自动开 PR、更新 ticket
- 处理不了的问题进 Inbox 等人类

**关键细节**：自动化触发的是一个 **skill**（`$skill-name`），而不是贴了一墙指令的 cron 任务。

### 2. Stripe's Minions：每周 1300+ PR

规模化的经典案例。

**触发**：在 Slack @Minion bot 或在消息上添加 emoji 反应——fire-and-forget。

**关键设计**：确定性编排器在 LLM 之前先工作：
1. 扫描消息中的链接
2. 拉取 Jira
3. 查找文档
4. 用 Sourcegraph + MCP 搜索相关代码

> 等到 LLM 登场时，所有需要的材料已经摆在桌上了。

**架构特点**：
- 6 层：确定性关卡和 LLM 创意步骤交替互锁
- Agent 写完代码 → 硬编码的 pipeline 跑 linter（agent 无法跳过）→ Agent 修复 lint → 硬编码 git commit
- 沙箱用 Devbox on EC2——"Cattle not pets"

> "AI 的可靠性来自约束的质量，而不是模型的大小。"

**人类没有离开，只是换了工位**：1300+ PR 仍然由工程师 review——时间从"写代码"转移到了"审查代码"。

### 3. 调度方案对比

| | Cloud Routines | Desktop 定时任务 | `/loop` |
|---|--------------|----------------|--------|
| 运行位置 | Anthropic 云 | 你的机器 | 你的机器 |
| 机器需开机？ | 否 | 是 | 是 |
| 会话需打开？ | 否 | 否 | 是 |
| 最小间隔 | 1 小时 | 1 分钟 | 1 分钟 |
| 能看到本地文件？ | 否（每次都 clone） | 是 | 是 |

**选择逻辑**：
- 高频 + 本地 → 用本地 `/loop`（代价：机器必须开机）
- 随机关机也能跑 → 用云端调度（代价：最小 1 小时间隔）

> "Local `/loop` 是'我在这时让它多跑几轮'。云端调度是'我不在时它也跑'。两回事，别混。"

## 七、四大成本 (§07)

> "一个无人看管的 loop 也是一个无人看管地犯错的 loop。"

| 成本 | 症状 | 防护一句话 |
|------|------|-----------|
| **Verification Debt** (验证债务) | 输出堆积无人验证，错误静默积累 | 安装一个不是干活的 agent 的评估器 |
| **Comprehension Rot** (理解腐烂) | 代码在增长，你脑中的地图停在三个月前 | 定期阅读输出；不能解释意味着该更新了 |
| **Cognitive Surrender** (认知投降) | 接受 loop 给的一切，不再有意见 | 执行可以外包，决策不行 |
| **Token Blowout** (Token 爆炸) | 用量剧烈波动，账单不可预测 | 上线前就设定好预算和重试上限 |

### Verification Debt (验证债务)

你让 loop 开 PR、改代码、合并——每一步都省了时间。但省下的时间变成了"还没被人验证过的输出"堆在那里等你还债。

问题在于债务不立刻找上门。Loop 改了二十个文件，测试全绿，PR 全开——看起来漂亮。问题藏在测试没覆盖的地方，在"能跑"和"正确"之间的缝隙里。它们静默积累，直到某个上线早上一起爆炸。

### Comprehension Rot (理解腐烂)

> "Loop 输送代码的速度越快，实际存在的和你实际理解之间的差距就越大。"

为什么发生？因为读代码比写代码无聊，而 loop 已经把所有的写都接管了。你再也不需要逐行读文件了，所以自然就不再读了。代码库在增长，你脑中的地图却停在三个月前。

最危险的是这个差距**在正常情况下不响警报**。项目在正常运行，你觉得理解得足够好——直到一个 bug 钻进你从没读过角落，loop 修不了，你不得不自己上手。

### Cognitive Surrender (认知投降)

> "当 loop 自己运行时，你很容易就不再有意见，接受它给的一切。"

验证债务和理解腐烂还是技术问题。认知投降是**态度问题**——不是"没时间"，而是"不想费事了"。Loop 跑得越稳，你越容易信任它；越信任它，越少去看；越少去看，越没有能力判断。一个自我加速的下坡。

### Token Blowout (Token 爆炸)

Loop 会自己生小帮手、自己重试、定时运行。你写的是逻辑，出去的是运行次数，次数乘以单价就是钱。一个 bug 让它空转一夜，你早上醒来看到的不是修好的代码，而是一张陌生的账单。

**防护**：上线前设定硬上限——per-run 预算、每日预算、最大重试次数。到达上限就停。

> loop engineering 最迷人的地方是让一个人做一个团队的工作。最危险的地方也是同一个点——一个团队会互相审视、互相提醒、互相争论。一个人加一堆 loop，很容易变成一个没人争论的回音室。

## 八、工程师的身份危机 (§08)

> "两个人构建同一个 loop，得到完全相反的结果。"

Loop 是一个系统，系统是中立的——同样的代码应该跑出同样的结果？但 Addy 说的不是 loop 本身，而是**站在 loop 外面的人**。

### 两种人

| | 第一种人 | 第二种人 |
|---|---------|---------|
| 如何使用 loop | 加速已经掌握的能力 | 逃避理解 |
| 能否读代码 | 能，有方向感 | 不能？让 loop 写 |
| 能否判断质量 | 能，loop 只是解放双手 | 不能？让 loop 合并 |
| 6 个月后 | 变得更强 | 成了自己不知道在跑什么的机器的看门人 |

> "Loop 不偏袒任何一方。它只是一个忠实的乘号，乘的是你。"

### 稀缺性的转移

Loop 让一件事变得极其便宜：**生成**。代码、计划、PR、修复——都可以被批量生产，几乎免费。

那什么仍然稀缺？**判断**。知道哪个计划是对的，哪行代码应该被阻止，哪个输出能跑但其实错了——这些是 loop 不能替你做的事。

> "Loop engineering 不贬低判断力的价值——恰恰相反。它拿走了所有不需要判断的工作，剩下的全是判断。"

### 更残酷的放大器

过去，一个错误判断最多让你手写了一小段错误的代码——影响有限，还有机会发现。

现在，一个错误判断会让 loop 忠实地、批量地、不知疲倦地执行那个错误一百遍。它不会停下来问"这个真的对吗？"——它只会跑得更快。

> "工具越强，对你的判断下的注就越大。你不能再指望'流程慢'来救你了，因为流程已经没有慢档了。"

### 两条路径

```diff
+ 第一种人：留下了一扇门，随时可以走进去看看里面在发生什么
- 第二种人：把门焊死了，赌自己永远不需要再进去
```

> **那天他们不得不进去的时候，发现自己已经没有钥匙了。**

### 结语

> **"Build the loop. But build it like someone who intends to stay the engineer, not just the person who presses go."**

这个选择不是建 loop 那天做一次就完了——**它每天都在被重新决定**。今天你多读了一个 PR，多问了一次"这个真的对吗？"，你还在工程师这一边。今天你走了捷径闭眼合并了，你就朝按钮那一侧滑了一格。没有人天生是工程师，也没有人能一次永远地做工程师。这是一个需要每天重新确认的身份。

## 九、构建你的第一个 Loop (§09)

### Five-step 入门

**Step 1: 跑一个 `/loop`**

最简单的起点，Claude Code v2.1.72+ 内置：

```
/loop 5m check the deploy    # 固定间隔
/loop check the deploy       # 让 Claude 自定节奏
/loop                        # 执行 .claude/loop.md 中的任务
```

注意：Session 内，7 天后过期；机器关了就停。

**Step 2: 让它可以读取 CI 和 Issues 做 Triage**

给一个 prompt 每天早上看三件事：CI 有没有失败、有没有新 Issue、最近改了啥。挑出值得处理的事列个清单。

**Step 3: 加一个状态文件让它有记忆**

Triage 结果写到 markdown 文件里（而不是留在聊天窗口里）。这个文件就是 loop 的 memory。

**Step 4: 加一个评估器让它能说不**

最关键也最容易跳的一步。用 `/goal`（v2.1.139+）：

```
/goal all tests in test/auth pass and the lint step is clean
```

判断完成条件的是另一个小模型，不是干活的 agent。

**Step 5: 加 worktrees 让它能并行**

Claude Code 用 `--worktree` 或 `-w` 为每个后台 agent 开独立的 worktree。

### First-loop 检查清单

| 元素 | 问自己 |
|------|--------|
| 发现源 | 它定时读什么？（CI / Issues / Commits / Inbox） |
| 状态文件 | 哪个磁盘文件存跨轮记忆？ |
| 评估器 | 有独立的检查能说不吗？ |
| 隔离 | 每个并行 agent 有自己 worktree 吗？ |
| Token 上限 | 设了花费上限吗？跑飞了谁来停？ |
| 人工审查点 | 哪一步会停下来让你看？ |

> 前两个决定 loop 能不能跑；后四个决定它跑起来会不会出问题。新手最常只带着前两个就上线了——结果是**一个没人看、没人能停的 loop，坐在那里反复点头认可自己**。
